🏕️ Lizetka editor — login & publish flow

From opening lizetka.cz/editor/ to a published post.

How the OAuth handshake works, what the Worker does, and which rules GitHub enforces.

🧑 Author's browser (static editor page) ☁️ Cloudflare Worker (lizetka-auth) 🐙 GitHub

A · Logging in (the OAuth handshake)

Happens once. After this the author stays logged in — tokens don't expire.

1
Browser

Author opens https://lizetka.cz/editor/. It's a plain static page — no server, no database. Loads instantly.

2
Browser

Author clicks “Login with GitHub.” The editor generates a random state string (a one-time anti-forgery token), saves it, and redirects the browser to GitHub's authorize page with the public client_id, the redirect_uri, scope=public_repo, and that state.

↓ browser is sent to github.com/login/oauth/authorize

3
GitHub

Shows the “Authorize Lizetka editor?” consent screen (only the first time). The author approves.

4
GitHub

Redirects the browser back to the editor with a temporary ?code=…&state=….

🔒 GitHub rule: the code is only ever sent to the pre-registered callback URL (https://lizetka.cz/editor/) — nowhere else.

5
Browser

The editor checks the returned state matches the one it saved (rejects the callback if not — CSRF guard), then strips code/state out of the URL bar.

6
Browser

The editor can't turn the code into a token itself — that step needs the secret, and GitHub blocks browser-to-token requests (no CORS). So it POSTs the code to the Worker instead.

↓ POST https://lizetka-auth.lizetka.workers.dev

7
Worker

Adds the client_id + the secret client_secret (which lives only here) to the code and calls GitHub's token endpoint.

↓ POST github.com/login/oauth/access_token

8
GitHub

Verifies the code + secret and returns the access token (scoped to public_repo).

🔒 GitHub rule: it's the temporary login code — not the token — that's single-use and expires within minutes, so a stolen code is useless. The token itself is long-lived (see step 10).

9
Worker

Hands only the token back to the editor (never the secret), with CORS limited to the allowed origins. Stores nothing — its job is done.

↑ returns { access_token, scope } to the browser

10
Browser

The editor saves the token in the browser (localStorage). The author is now logged in — and stays logged in across reloads, since these tokens don't expire.

B · The access check (who's allowed to publish)

Login proves identity. Permission to publish is a separate gate — the repo's collaborator list.

11
Browser

The editor asks GitHub GET /repos/kratoon/lizetka and reads permissions.push. If the user isn't a collaborator, they land on a friendly “no access” screen — the publish UI never appears.

12
GitHub

Confirms whether this user has Write (collaborator) access to kratoon/lizetka. This list — managed in the repo's Settings → Collaborators — is the real allow-list of who can publish.

C · Publishing a post

One atomic commit straight to main from the browser — which triggers the site to redeploy.

13
Browser

Author writes the post and clicks Publish. The editor builds one atomic commit (post JSON + any PDFs) via GitHub's Git Data API, sending the token in every request: Authorization: Bearer <token>.

↓ commit to main

14
GitHub

Accepts the commit only because the user has write access — it would reject anyone else server-side, no matter what the browser tried. The commit is attributed to the actual author. Landing on main triggers the CI action → the site rebuilds and deploys to lizetka.cz. ✅

🐙 What GitHub enforces

  • Sends the login code only to the pre-registered callback URL.
  • Requires the client_secret to trade a code for a token (so a browser alone can't).
  • The code is single-use and short-lived.
  • The token is scoped to public_repo — it can only touch the user's own public repos.
  • Rejects commits from non-collaborators server-side — the real, unbypassable access gate.
  • Attributes each commit to the actual logged-in author.

☁️ What the Worker does

  • Holds the client_secret — the one place it exists, never in the browser.
  • Performs the steps a static page can't (they need the secret): the code → token exchange at login…
  • …and revoking the token on logout — a kill switch that invalidates it on GitHub, killing even a leaked copy.
  • Returns only the token; never echoes the secret.
  • Limits CORS to the editor's known origins.
  • Does NOT: store data, see passwords, or decide who's allowed to publish — that's the repo's collaborator list.
💡 Logging out revokes the token. Clicking “Odhlásit a odvolat přístup” in the editor tells GitHub to delete the token entirely — so a leaked copy dies instantly. Logging out when you're done is the recommended habit.
The Worker is the system's only backend — there for the two steps that need the server-held secret a static page can't keep: exchanging the login code for a token, and revoking that token on logout. Access control is entirely GitHub's repo collaborator permission.
